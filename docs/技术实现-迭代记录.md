## 第三轮 (2026-05-15)

### 1. 模板编辑器 + 历史数据重处理

**需求：** 用户修改导入模板后，可以选择"更新所有历史数据"，将新的字段映射应用到之前导入的文件。

**前提条件：**
- 原始 Excel/CSV 文件已在导入时保存到 `/app/uploads/{file_id}`，不会删除
- `ImportSession` 表有 `template_id` 字段，关联到 `ImportTemplate`
- `merge_engine.upsert_transactions` 天然支持 upsert，重处理不会产生重复

**API：**

| 方法 | 端点 | 说明 |
|------|------|------|
| `PUT` | `/settings/templates/{id}` | 更新模板，不触发重处理 |
| `POST` | `/settings/templates/{id}/reapply` | 异步重处理所有关联的 import session |
| `POST` | `/settings/templates/save-as` | 另存为新模板，不对历史数据生效 |

**重处理流程：**
```
POST /templates/{id}/reapply
  → 查找 ImportSession WHERE template_id = id AND status = 'completed'
  → 若 0 个 session：直接返回 {task_id: "", total_sessions: 0}
  → 若 > 0 个：启动后台 asyncio Task，逐 session：
      1. 从磁盘读取原始文件
      2. 用新 field_mapping + 原有 account_binding 重新解析每一行
      3. upsert_transactions() — 按 external_tx_id / amount+type+date 匹配后更新
      4. match_refunds() — 重新匹配退款
      5. 更新进度到 TaskManager
```

**前端：** `TemplateEditor.tsx` 弹窗 — 三个按钮：
- "保存并更新历史数据"：PUT 模板 → POST reapply → 显示 ProgressBar
- "另存为新模板"：POST save-as → 关闭
- "取消"

### 2. 统一异步任务 + 进度展示

**需求：** 导入和模板重处理都可能耗时数十秒，需要后台执行 + 实时进度条。

**后端 TaskManager** (`backend/app/services/task_manager.py`)：
- 线程安全的内存字典，key 为 UUID
- `create(type, total)` → task_id
- `update(task_id, current)` — 更新进度
- `complete(task_id, result)` — 标记完成，附带统计
- `fail(task_id, error)` — 标记失败
- `get(task_id)` → 当前状态
- 完成后 10 分钟自动清理过期任务

**API：**

| 方法 | 端点 | 说明 |
|------|------|------|
| `GET` | `/tasks/{task_id}/progress` | 返回 `{type, current, total, status, result, error}` |

**导入执行改造：**
- `POST /import/execute` 不再等待完成，立即返回 `{task_id, total}`
- 后台 `asyncio.create_task()` 逐批处理（每 50 行一批），每批更新进度
- 完成后 `task_manager.complete()` 写入 created/updated/matched_refunds 统计

**前端 ProgressBar** (`frontend/src/components/ProgressBar.tsx`)：
- 每秒轮询 `GET /tasks/{task_id}/progress`
- 三种状态：进度条（processing）→ 绿色摘要（done）→ 红色错误（error）
- 在 `ImportPage` 导入阶段和 `SettingsPage` 重处理时使用

### 3. 账户余额自动更新

**问题：** 导入流水线创建交易后，`account.current_balance` 不更新，用户必须手动点"重新计算"。

**Root Cause:** `merge_engine.upsert_transactions` 只向数据库写入 Transaction 行，从未调用 `balance_calculator.apply_balance()`。

**修复：**
- `merge_engine.py`：新交易创建后调用 `await apply_balance(session, tx)`
- 更新场景（`_merge`）只修改 merchant_name、description 等元数据，不影响余额，无需调用

---

### 4. 转入交易的余额计算修复

**问题：** 账户详情页的流水列表和余额重算都不包含"转入"交易（transfer_account_id = 本账户）。

**修复：**

**重算端点** (`POST /{id}/recalculate`)：
- 原查询：`SUM(CASE ...) WHERE account_id = account_id`
- 新增查询：`SUM(base_amount) WHERE transfer_account_id = account_id AND type = TRANSFER`
- 最终余额 = initial_balance + 本账户交易影响 + 转入金额

**交易列表** (`GET /{id}/transactions`)：
- 原查询：`WHERE account_id = account_id`
- 新增查询：`WHERE transfer_account_id = account_id AND type = TRANSFER`
- 合并后按日期倒序排列
- 向后遍历时区分"转出"（running += amount）和"转入"（running -= amount）

### 5. 微信交易类型拼接到商户名

**需求：** 导入时把 `source_category`（如"商户消费"）拼接到 `merchant_name` 前面。

**实现** (`merge_engine.py` line 39-46)：
- 两者都有值：`"商户消费 · 美团"`
- 只有 source_category：直接用 source_category 作为 merchant_name

### 1. Docker 卷路径修复

**问题：** `docker-compose.yml` 中 `./data:/app/data` 相对路径从 WSL 调用时无法正确映射到 Windows Docker daemon，导致容器获得空数据库。

**修复：** `docker-compose.yml` 卷路径改为 `${HOST_PROJECT_ROOT:-.}/data`，`.env` 新增 `HOST_PROJECT_ROOT=D:\work\where_is_money`。

**问题：** 预设导入模板（微信支付）未持久化，与 TagRule 的系统默认规则设计不一致。

**修复：** `routers/settings.py` 新增 `seed_preset_templates(db)`，在 GET /templates 时自动创建微信支付预设模板：

```
交易时间→transaction_date, 交易类型→source_category, 交易对方→merchant_name,
商品→description, 收/支→type, 金额(元)→original_amount,
当前状态→transaction_status, 交易单号→external_tx_id,
商户单号→merchant_order_id, 备注→remark
```

同时修改 `models/import_template.py` — `user_id` 改为 nullable（预设模板不需要绑定用户），`main.py` lifespan 中检测并重建旧表。

### 3. 导入分类进度显示
