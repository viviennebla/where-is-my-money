# 智能导入与编辑优化 — 设计方案

## 需求背景

用户在 v0.4 使用中提出三个优化点：

1. 导入时缺少基于关键词的本地标签匹配（不依赖 AI）
2. 流水列表中的内联编辑与实际使用场景不符（Excel 导入的数据不应随手改）
3. 重复导入时，新文件可能包含旧记录缺失的字段，应提示用户选择是否补充

---

## 需求 1：可配置的关键词标签匹配规则

### 问题

当前导入后自动分类依赖两条路径：
- 本地规则：同一 `merchant_name` 查找历史标签
- AI 兜底：调用 DeepSeek 推断（需要 API Key）

缺陷：
- 关键词规则硬编码在代码中，用户不可见、不可改
- 未配置 API Key 时新商户无法分类
- AI 调用有延迟和成本
- 用户不知道自己导入了什么规则，也无法按需调整

### 方案

**设计思路与列映射（ImportTemplate）一致：**
1. 系统预置一套默认规则（类似 preset template）
2. 规则存储在数据库中，用户可见、可增删改
3. 导入时按规则匹配，不依赖 AI
4. 规则可绑定到特定字段（交易对象、描述、来源分类）

### 数据模型

**新增 `TagRule` 表：**

```python
class TagRule(Base):
    __tablename__ = "tag_rules"

    id: str (PK, UUID)
    user_id: str | None (FK → users, NULL = 系统默认)
    field: str  # 匹配字段: "merchant_name" | "description" | "source_category"
    keyword: str  # 匹配关键词
    tag_id: str (FK → tags)
    is_system_default: bool (default=False)
    created_at: datetime
    updated_at: datetime
```

### 匹配逻辑

```
1. 本地规则：同 merchant_name → 用户历史标签
      ↓ 未命中
2. 标签规则引擎：遍历用户规则 + 系统默认规则
   - 字段匹配 + 关键词包含匹配 → 对应标签
      ↓ 未命中
3. AI 兜底：DeepSeek 推断（需要 API Key，可选）
```

## 需求 2：禁用列表内联编辑

### 问题

当前在流水列表行中点击"交易对象"或"描述"会弹出 input 框允许直接编辑。用户认为这些数据来自 Excel 导入，在列表中随手改动不合适，编辑应进入详情页（TransactionEditor 弹窗）进行。

### 方案

- 移除 `TransactionRow` 中 `merchant_name` 和 `description` 的点击编辑逻辑
- 单元格改为纯文本展示
- 仅保留"编辑"按钮（打开 TransactionEditor 弹窗）

### 文件改动

- `components/transactions/TransactionRow.tsx` — 移除 `startEdit`/`saveEdit`/`editingField`/`handleKeyDown` 逻辑，`<td>` 不再有 `cursor-pointer` 和 `onClick`

## 需求 3：合并引擎字段补充

### 问题

当前 `_merge()` 遇到旧记录已有值、新记录也有值时，静默选择更长的字符串（`_non_null`）。但用户可能不希望自动覆盖。

实际场景：
- 第一批导入只有金额和日期，没有交易对象
- 第二批导入来自另一个渠道（如支付宝），同一笔交易有商户名和描述
- 用户希望能看到差异并决定是否用新数据补充旧记录

### 方案

**安全合并策略：仅补充空白字段，已有值不覆盖。**

- `new_val` 有值 且 `old_val` 为空 → 填充（安全补充）
- `new_val` 有值 且 `old_val` 也有值 且新值更丰富 → 跳过，记录到 diffs
- 导入完成后前端展示差异，用户可审核

### 后端

`_merge` 改为仅补充策略：

```python
async def _merge(session, existing, rec) -> list[dict]:
    """Returns diffs for fields where new is richer but old already has value."""
    diffs = []
    for field in ("merchant_name", "description", "remark", ...):
        new_val = rec.get(field)
        old_val = getattr(existing, field, None)
        if new_val and not old_val:
            setattr(existing, field, new_val)  # 安全补充
        elif new_val and old_val and len(new_val) > len(old_val):
            diffs.append({"transaction_id": existing.id, "field": field, "old": old_val, "new": new_val})
    return diffs
```

### 前端

- 导入结果页展示差异摘要表格（字段、当前值、新值）
- 暂不实现交互式审核（后续迭代）

### 文件改动

| 文件 | 说明 |
|------|------|
| `services/merge_engine.py` | `_merge` 改为仅补充策略 + 返回 diffs |
| `routers/import_pipeline.py` | 传递 diffs 到 task result |
| `components/ProgressBar.tsx` | 展示 diffs 数量 |
| `components/import/ImportProgress.tsx` | 传递 diffs |
| `pages/ImportPage.tsx` | 展示差异摘要表 |

## 需求 4：账户详情页复用流水列表

### 问题

`AccountDetailPage` 有自己独立的交易表格实现：无分页、无筛选、字段不一致（缺标签列、缺账户列、多了余额列），与主页体验割裂。

### 方案

- `TransactionList` 接受 `fixedAccountId` prop，锁定到指定账户
- `TransactionFilters` 接受 `hideAccountFilter` prop，隐藏账户下拉
- `AccountDetailPage` 只保留账户头部信息，下方直接复用 `TransactionList`

### 文件改动

| 文件 | 说明 |
|------|------|
| `components/transactions/TransactionList.tsx` | 新增 `fixedAccountId` prop |
| `components/transactions/TransactionFilters.tsx` | 新增 `hideAccountFilter` prop |
| `pages/AccountDetailPage.tsx` | 移除内联表格，复用 TransactionList |
