Where is my money (WIMM) - 产品需求规格说明书 (PRD)
0. 产品定位与目标
产品名称：Where is my money (WIMM)

核心痛点：多平台（微信、支付宝、各银行）账单分散且格式不一；传统记账软件手动录入繁琐，且存在隐私数据被平台收集的风险。

解决方案：基于云端部署、本地优先（Local-First）的个人记账系统。主打 “大模型驱动的文件解析 + 多源数据智能合并”，实现账单的高效导入、自动分类与资金流向精准追踪。

目标端侧：响应式 Web 端（兼容 PC 与 Mobile）+ PWA（支持离线浏览）。

1. 核心业务范围 (Scope)
1.1 交易记录管理 (Transactions)
全场景覆盖：支持消费（支出）、收入、转账（给他人）、退款。

多币种支持：

每笔交易记录 original_currency（原币种）和 original_amount（原金额）。

系统设定全局主币种（如 CNY），并记录折算后的 base_amount 以供报表统计。

退款全生命周期闭环：

强关联：退款记录必须且只能通过 parent_id 关联一笔历史“支出”或“转账”。

余额联动：退款发生时，原路增加对应资金账户的余额。

统计逻辑：报表统计时，实际支出 = 原支出金额 - 关联的退款金额。

1.2 账户体系 (Account Management)
账户定义：记录资金存放处的实体（如：招商银行、微信零钱、支付宝）。包含 initial_balance（期初余额）和 current_balance（当前推算余额）。

内部划转 (Internal Transfer)：

定义：资金在本人拥有的两个账户间流动（例如从招行提现到微信）。

统计与约束：必须指定 transfer_account_id（目标账户）。内部转账绝对不计入全局的总收入和总支出，仅更新转出和转入账户的双边余额。

2. 核心创新点：智能数据导入工作流 (Data Pipeline)
整个导入链路拆分为四个标准步骤：

步骤 A：文件上传与预处理
支持主流平台的 CSV 和 Excel 导出文件。

提取文件的前 5 行（包含表头及样本数据），准备进入推断流程。

步骤 B：AI 智能推断与人工验证 (Human-in-the-loop)
推断：调用大模型（DeepSeek）分析文件头，推断原文件列名与系统标准字段的映射关系。

标准字段：交易时间、金额、收支方向、商户名称、外部交易单号、商品描述等。

验证与记忆：前端向用户展示推断映射，用户可手动调整。确认后，系统将该映射持久化为特定平台（如“微信账单”）的“导入模板”（Template），后续复用。

步骤 C：多源数据合并与信息富化 (Merge & Enrich)
系统按模板解析全量文件，并在落库时执行 Upsert（更新插入）逻辑。

匹配规则（去重防碰撞）：

强匹配：若两笔记录拥有相同的“外部交易单号”（external_tx_id），直接命中。

模糊匹配（时间窗算法）：对于无单号记录，若 “金额完全相同” + “收支方向一致” + “交易时间误差在 ±24 小时内”，判定为同一笔交易。

富化策略：命中同一笔交易时，系统合并非空字段。若描述产生冲突（如微信的“商户名” vs 京东的“商品明细”），优先保留业务侧的详细明细，资金侧的描述降级存入 remark 或保留原始 JSON 备查。

步骤 D：受约束的 AI 智能分类 (Constrained Classification)
分类体系：采用灵活的“标签（Tag）”模式。系统预置基础标签树（如：餐饮、交通、居住、娱乐、医疗、转账等，标记为 is_system_default）。

分类执行逻辑：

本地规则优先：优先根据系统内已有的学习规则（如用户曾将“星巴克”手动划为“餐饮”）进行匹配。

AI 兜底：未命中规则时，调用 DeepSeek 进行分类。强制约束大模型优先从“预设标签库”中选择；仅当无任何符合项时，才允许生成长度 ≤4 字符的新标签。

3. 本地优先与离线能力 (Local-First & Offline)
本地存储：前端集成 IndexedDB（如 Dexie.js），结构化存储用户的账单流水与账户状态，确保在断网环境下系统依然可用、可读。

状态展示与同步机制：

前端全局状态维护两个时间戳：last_local_update（本地数据最后修改时间）与 last_sync_success（上次成功与云端同步的时间）。

UI 要求：在应用的导航栏或状态栏显式展示“上次同步时间：X分钟前”，并提供手动重试同步的交互按钮。

4. 系统模块与安全规范 (Architecture & Security)
4.1 技术栈映射
前端：React/Vue + TailwindCSS + IndexedDB + PWA (Service Worker 缓存静态资源)。

后端：FastAPI + SQLAlchemy + SQLite/PostgreSQL。部署于自建云服务器的 Docker 容器中。

4.2 API 与 AI 交互约束
身份鉴权：强制要求后端接口具备基础鉴权（JWT），拒绝匿名访问。

隐私脱敏：上传给 DeepSeek 进行推断和分类的数据，仅限表头、商户名称及描述（可选脱敏金额），绝对禁止将整个用户的原始 CSV 账单文件上传至大模型 API。

结构化输出强制性：任何调用大模型的 API 封装，必须在 Prompt 中强制要求返回严格的 JSON Schema 格式，避免解析崩溃。

5. MVP 版本（Phase 1）验收标准
作为开发的第一阶段，需优先交付以下链路：

数据库：完成 Transaction, Account, Tag, ImportTemplate 的表结构搭建（包含所有核心字段及外键关联）。

核心后端 API：实现基础 CRUD、多源数据合并引擎的 Upsert 逻辑、资金余额的加减计算。

大模型集成：跑通 “发送样例数据 -> DeepSeek 推断 -> 成功返回 JSON 映射关系” 的接口层逻辑。

前端列表视图：实现可筛选的交易流水列表，并能手动编辑修改分类标签。
